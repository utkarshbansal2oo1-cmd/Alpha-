# AlphaSource Product Strategy

**The product bible.** Everything below assumes the technical foundation already built and approved across Sprints 1-10 (Query Understanding, Knowledge Engine, Search Planner, Candidate Repository, Discovery Engine, Evidence Lake, Autonomous Discovery). This document does not revisit or re-justify any of that architecture — it answers the business questions a CEO, CTO, CHRO, or investor will ask about it.

---

## 1. Vision

Recruiters currently split their attention across five or six disconnected systems — LinkedIn Recruiter, Naukri, an ATS, spreadsheets, referral threads — manually reconciling all of it into one picture of "who should I talk to." AlphaSource's vision is not to be a seventh tab. It is to be the layer that sits *above* all of them: the place a recruiter describes what they need once, in plain English, and gets an explainable, continuously-improving answer regardless of which underlying sources that answer came from.

The long-term version of this vision — already staged architecturally in Sprints 8-10 — is a system that is not dependent on any single data provider surviving, changing its terms, or raising its prices. That independence is not a defensive feature. It is the entire commercial thesis: every competitor in this space (see Question 5) is structurally dependent on LinkedIn's data, LinkedIn's API access, or a narrow set of owned data. AlphaSource's Evidence Lake architecture is the only design in this list of competitors, based on public information available, that treats every source as replaceable evidence rather than as the product itself.

## 2. Question 1 — Where Does Candidate Data Come From?

| **Research Publications** | Public | Strong, hard-to-fake signal of deep expertise for research/technical roles | Narrow applicability — irrelevant for most roles | Medium — requires parsing academic metadata (already anticipated in Sprint 9's evidence types) | Low-medium — valuable for a specific, high-value candidate segment (senior technical/research hires) | Low — published, public work |
| **Conference Speakers** | Public | Strong seniority/expertise signal (per Sprint 9's 85% baseline confidence), corroborates other evidence well | Sparse — most candidates never speak at a conference | Medium — per-conference-site parsing effort | Low-medium alone; valuable as corroboration | Low — public speaker listings |
| **Public APIs** (general) | Public / Partner (varies) | Extensible catch-all category — new sources can be added without new architecture (per Sprint 8's Connector Framework design) | Quality and availability vary enormously per API | Varies per API | Varies per API | Must be evaluated individually — "public API" does not automatically mean "compliant to use for this purpose" |

**The structural takeaway feeding directly into Question 3**: the sources with the strongest combination of technical feasibility, commercial feasibility, and low compliance risk — ATS, resume upload, CSV, internal DB, GitHub — are exactly the ones already prioritized first in Sprint 8's and Sprint 9's own implementation roadmaps. That was not a coincidence; it reflects the same judgment restated here in commercial terms.

## 3. Question 2 — Acquisition Strategies

| Strategy | How it works | Why it matters |
|---|---|---|
| **Customer uploads resumes** | Bulk or one-at-a-time upload into the Evidence Lake via the Resume Connector | Zero-friction, works for any customer regardless of existing tooling maturity — the fastest path to a customer's first real search results |
| **Customer connects ATS** | OAuth/API-key connection to Greenhouse, Ashby, etc. via the ATS Connector | Turns a customer's existing applicant history into immediately-searchable evidence — highest-perceived-value integration for a customer already invested in an ATS |
| **Customer connects recruiter accounts (where supported)** | Only where a platform offers an *official, authorized* account-level integration (not scraping a personal login) | This is explicitly narrow — most platforms (LinkedIn, Naukri) do not offer this today; the architecture must never assume it will become available, and must remain fully valuable without it (per Sprint 9's compliance section) |
| **Public discovery** | The Autonomous Discovery System (Sprint 10) proactively seeking evidence from GitHub, career pages, conference pages, etc. | This is what makes AlphaSource's Evidence Lake grow *without* the customer doing manual work — the core differentiator versus "you must feed us everything" competitors |
| **Recruiter browser extension** | A lightweight extension letting a recruiter capture a profile they're already viewing (on any site they have legitimate access to) into the Evidence Lake as a "recruiter-sourced" evidence item | This is the single most important acquisition strategy for the LinkedIn/Naukri problem: it puts the *compliance burden on the human's own authorized session*, not on AlphaSource scraping anything — the recruiter is looking at a page they're legitimately logged into and choosing to save it, the same way copying a profile into a spreadsheet today works, just faster and structured. This deserves priority build attention in Version 1/2 (Question 3). |
| **Email forwarding** | A recruiter forwards a candidate email (an inbound resume, a referral intro) to a dedicated AlphaSource address, which parses and ingests it | Fits naturally into how recruiters already work (email is still the dominant candidate-communication channel) — low-friction, no new habit to build |
| **Manual import** | Structured import UI for one-off cases the CSV connector doesn't cleanly fit | A fallback, not a primary strategy — exists so no legitimate customer data is ever un-importable |
| **Scheduled sync** | Recurring, automated re-pull from any connected source (ATS, HRMS), per Sprint 8/9's freshness mechanics | Keeps connected sources current without a recruiter remembering to manually re-sync |
| **Future partnerships** | Formal, authorized data-sharing agreements with platforms willing to offer one (a Wellfound-style partnership, a regional job board) | Upside optionality, never a dependency — every acquisition strategy above this row must remain sufficient on its own, so a partnership is a bonus tier, not a prerequisite |

## 4. Question 3 — Commercial Strategy

**Version 1 should include**: Resume upload, CSV import, ATS connector (starting with one or two of the most common — Greenhouse and Ashby are reasonable first targets given their API maturity per the pricing research above), GitHub connector, and the recruiter browser extension. This set covers Company Owned, Candidate Owned, and compliant Public sources entirely — zero Restricted-source dependency — while already being differentiated (no direct competitor combines "plain-English search" with "explainable match reasoning" over this source combination, per Question 5).

**Version 2 should include**: Company HRMS connectors (for enterprise internal-mobility use cases), conference/publication/patent evidence sources (Sprint 9's higher-context, lower-volume evidence types — valuable once there's enough Version 1 volume to meaningfully corroborate against), email forwarding, and the first Autonomous Discovery capabilities from Sprint 10 (starting with Scheduled Refresh and Background Discovery once there's a stable Version 1 customer base generating real Mission-outcome data to learn from).

**What should never be built**: any connector or acquisition path that depends on bypassing a platform's authentication or terms of service — this is not a phased-later item, it is permanently out of scope, restated here exactly as it was constrained architecturally in Sprints 8-10. Concretely: no LinkedIn scraping, no Naukri scraping, no credential-sharing "log in as the recruiter" browser automation against platforms that prohibit it in their terms.

**What should require partnerships**: LinkedIn, Naukri, and any platform-level "connect your recruiter account" integration beyond what a browser extension capturing a legitimately-viewed page can achieve. These remain aspirational, not planned, until a specific partnership exists.

**What should require customer authorization**: every Company Owned source (ATS, HRMS, internal DB) and every Candidate Owned source (resumes, referrals) — authorization here means the customer's own consent/data-processing agreement, standard for any B2B SaaS integration, not a special case.

## 5. Question 4 — Product Positioning

| Position | What it implies | Fit for AlphaSource |
|---|---|---|
| Search Engine | "You type a query, we return results" — narrow, commoditized | Too narrow — undersells the Knowledge Engine/explainability work already built, and is the position most vulnerable to being replicated by anyone with an LLM and a database |
| Sourcing Tool | A tool recruiters use as one step in their existing process | Undersells the vision — implies AlphaSource is subordinate to LinkedIn/Naukri as a supplement, not a replacement for how a recruiter starts their day |
| Recruiter Copilot | An AI assistant embedded in a recruiter's existing workflow | Closer, but "copilot" framing implies a human is doing the primary work and AI is assisting at the margins — undersells the autonomous evidence-gathering (Sprint 10) that runs independent of any specific recruiter action |
| **Talent Intelligence Platform** | A system that understands, explains, and continuously improves its knowledge of the talent market — not just a tool, a knowledge layer | **Recommended.** This is the only framing that matches what's actually been built: a Knowledge Engine, an Evidence Lake, explainable matching, and autonomous, continuously-improving discovery. It is also the exact category Eightfold has already established and validated commercially (see Question 5) — proof the market will pay category-leading prices for this positioning specifically, not merely tolerate it. |
| Recruiting Operating System | The single system all recruiting activity runs through, replacing the ATS itself | Overreaches for where the product is today — this is a plausible Year 3+ destination (matching Sprint 10's "AI Hiring Operating System" long-term vision), not a Version 1/2 claim; claiming it prematurely invites a comparison against Greenhouse/Ashby's structured-hiring maturity that AlphaSource cannot yet win |

**Recommendation: Talent Intelligence Platform**, explicitly not an ATS replacement and not a LinkedIn replacement — a layer that makes the recruiter's existing sources (including LinkedIn/Naukri manually, via the browser extension, and everything else automatically) smarter, explainable, and durable regardless of any one source's future.

## 7. Question 6 — Business Model

**Recommended model: hybrid seat + usage, not pure per-seat.**

Rationale, directly informed by the pricing research above: every pure per-seat competitor (LinkedIn Recruiter, HireEZ) creates a specific customer pain point — recruiters pay full price whether they search once or a hundred times that month, and InMail-style overage fees (LinkedIn) are a well-documented source of customer frustration. AlphaSource's cost structure is also genuinely usage-correlated (LLM calls, evidence discovery cost per Sprint 10's Evidence Economics) in a way pure per-seat pricing doesn't reflect. A hybrid model — a base per-seat fee covering unlimited *reading* (viewing existing candidate intelligence, browsing the Talent Graph) plus metered pricing for *discovery-triggering* actions (new searches that invoke fresh AI understanding + evidence discovery) — prices the product in line with its actual cost drivers, which most competitors in this table do not do transparently.

**Concrete structure**:
- **Per-seat base fee**: access to the platform, unlimited search *against already-discovered* evidence (no new Discovery Engine cost incurred).
- **Per-search / AI usage pricing**: a metered charge for searches that trigger new Query Understanding + Discovery Engine work — priced to reflect real LLM/discovery cost (Sprint 10's cost model), not an arbitrary markup.
- **Connector pricing**: each connected source (ATS, HRMS) priced modestly per connection, reflecting integration/maintenance cost — not a primary revenue driver, more a mechanism to keep low-value "connect everything just because" behavior in check.
- **Enterprise licensing**: for large customers, a negotiated flat annual license bundling seats + a usage allowance, matching how Greenhouse/Ashby/Eightfold all ultimately price their largest accounts (custom-quoted, headcount-informed) — the market has already validated that large enterprise buyers prefer this over metered billing at scale.
- **Premium Intelligence tier**: an add-on for Talent Graph-level queries (Sprint 9 section 6's Candidate↔Candidate, Company↔Skill market-intelligence capabilities) — priced separately because this is a distinctly higher-value capability than basic search, matching how SeekOut's managed-sourcing add-on and Eightfold's agentic screening features are both sold as premium layers on top of a base platform.

**Per-company vs. per-recruiter**: base access should be per-recruiter-seat (matching market expectation and making individual usage attributable), but the *discovery/AI usage* component should be pooled at the company level — a shared usage allowance the company's admin can monitor and top up, avoiding the LinkedIn-style experience of an individual recruiter unexpectedly hitting a personal cap mid-search.

## 8. Question 7 — Risk Analysis

**Technical risks**: the Autonomous Discovery System's Learning Loop (Sprint 10) could degrade discovery efficiency if under-tuned — mitigated by the hard architectural boundary already established (learning affects prioritization only, never evidence correctness). Connector maintenance burden grows linearly with source count (Sprint 8) — mitigated by prioritizing structured-API sources (ATS, GitHub) over fragile HTML-scraping sources (generic career pages) in the acquisition roadmap.

**Legal risks**: the single largest legal risk in this entire strategy is any acquisition path that touches LinkedIn or Naukri outside officially authorized channels — explicitly excluded throughout this document and the prior architecture sprints, and this exclusion should be treated as a standing legal-review gate on any future connector proposal, not a one-time decision. Candidate data retention/consent (resumes, CSV uploads) carries jurisdiction-specific compliance obligations (flagged, not resolved, in Sprint 9 section 11) that need real legal input before Version 1 ships to any regulated market.

**Commercial risks**: being under-differentiated against Eightfold in enterprise deals where "talent intelligence platform" is already an established, trusted category Eightfold owns — mitigated by explicitly not chasing Eightfold's 10,000+-employee-only segment in Version 1/2, instead winning the mid-market Eightfold structurally ignores (Question 5). Pricing complexity (the hybrid model in Question 6) risks confusing early customers if not packaged simply at the point of sale — mitigated by presenting a single simple number for most customers ("$X per seat per month, includes Y searches") with the metered/enterprise complexity handled behind the scenes.

**Scalability risks**: covered extensively in Sprints 8-10's own scalability sections — the business risk specifically is under-provisioning Discovery/Evidence infrastructure ahead of a large enterprise customer's actual data volume; mitigated by the tiered, sequential scaling approach already designed (never build for 10M candidates before there's real 1M-candidate demand).

**How AlphaSource avoids dependence on any one platform** (the question underlying this entire document): structurally, via the Evidence Lake's provider-independence (Sprint 9) and the Discovery Engine's connector-agnostic routing (Sprint 8) — and commercially, by ensuring Version 1's source mix (Question 3) already delivers full product value using zero Restricted sources, so no customer's day-one experience of AlphaSource ever depends on LinkedIn or Naukri access existing at all.

## 10. Question 9 — Executive Pitch Narrative

**For the CEO**: "Every recruiting tool you've bought so far makes you more dependent on someone else's data — LinkedIn's, Naukri's. AlphaSource is the first system designed explicitly so that dependency doesn't exist. It gets smarter every day on its own, using data your company already has plus what it discovers automatically and compliantly — and it never stops working if one source changes its terms or its price."

**For the CTO**: "This isn't a wrapper around an LLM. It's a typed, tested pipeline — Query Understanding, a weighted Knowledge Graph, an Evidence Lake that never overwrites a fact, and an Identity Resolution system staged specifically so no AI decision is a black box. Every match is explainable back to the exact evidence that produced it. And architecturally, adding a new data source never requires touching the retrieval or intelligence layers — that's not a claim, it's how the interfaces are built."

**For the CHRO**: "Your recruiters currently spend most of their time searching, not talking to candidates. AlphaSource collapses that search time to one sentence, and shows its work — so you can trust a recommendation without needing to independently re-verify it. And because it plugs into AlphaRecrewt, the candidates it finds flow straight into assessment and interview, not into another disconnected tool."

**For Investors**: "The sourcing/talent-intelligence market has an obvious ceiling problem: every major player (LinkedIn Recruiter, HireEZ, SeekOut) is fundamentally a rent-seeking layer on top of LinkedIn's own data, priced accordingly (~$10-15K per seat per year at the top end). Eightfold proved the 'talent intelligence platform' category is worth pursuing — $2.1B valuation, enterprise contracts up to half a million dollars a year — but positioned it exclusively for 10,000+-employee companies, leaving mid-market completely uncontested. AlphaSource's Evidence Lake architecture is a structurally different bet: provider-independent by design, not by marketing claim, which is defensible in a way 'we have a good LLM prompt' is not, and it's aimed at the market segment the current category leader has explicitly chosen not to serve."

**Problem → Vision → Solution → Business Value → ROI → Future**, in one paragraph each, is the shape every one of the four narratives above already follows — problem (dependency/fragmentation), vision (a provider-independent intelligence layer), solution (the specific architecture built across Sprints 1-10), business value (time saved, explainability, resilience), ROI (fewer per-seat dollars spent on redundant tools, faster time-to-shortlist), future (Sprint 10's Evidence Lake → Talent Graph → AI Hiring Operating System progression, ending in the AlphaRecrewt handoff).

## 11. Question 10 — The Most Important Question

**Why would a recruiter open AlphaSource before opening LinkedIn Recruiter or Naukri Recruiter?**

> Because AlphaSource already knows what LinkedIn and Naukri would each show separately, plus what neither of them can show at all — and it tells you exactly why every name on the list is there before you spend a single InMail or credit finding out yourself.

This holds up under refinement because it doesn't compete with LinkedIn/Naukri on data volume (a fight AlphaSource cannot win head-on in Version 1/2) — it competes on being the *first stop that makes the second and third stops unnecessary, or at least much shorter*, by combining what a recruiter's own company already owns (ATS, resumes, referrals) with what's compliantly discoverable elsewhere (GitHub, evidence-based discovery), explained well enough that the recruiter trusts the answer instead of re-checking it manually. If a future version secures real LinkedIn/Naukri partnership access, this answer only gets stronger, never invalidated — provider-independence was never a bet *against* those platforms, it's what keeps the answer true whether or not that access ever arrives.
